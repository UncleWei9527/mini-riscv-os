# RISC-V 特权架构与中断控制体系：核心寄存器与指令全景图

> **写在前面的话**：RISC-V 的精髓在于“极简”与“权力隔离”。普通计算交给基础指令集，而操作系统的核心控制权（进程调度、权限降级、中断路由）全部由一组极其精妙的**控制与状态寄存器 (CSR, Control and Status Register)** 和少数几条**特权指令**来实现。

---

## 一、 核心特权指令 (Privileged Instructions)

在 RISC-V 中，操作特权寄存器和进行模式切换，必须依赖专属指令。普通 U-Mode 程序执行这些指令会直接触发非法指令异常。

| 指令 | 英文全称 | 动作大白话 | 操作系统实战场景 |
| :--- | :--- | :--- | :--- |
| **`csrr`** | CSR Read | **读 CSR**：把特权寄存器里的值读到普通通用寄存器里。 | `csrr a0, scause` (接线员读取案发原因) |
| **`csrw`** | CSR Write | **写 CSR**：把通用寄存器里的值强行写入特权寄存器。 | `csrw sepc, a0` (修改返回地址，实现时光倒流) |
| **`ecall`** | Environment Call | **主动求救**：主动触发一个异常，向更高特权级请求服务。 | U-Mode 的 App 想打印字符，执行 `ecall` 呼叫 S-Mode 内核。 |
| **`mret`** | Machine Return | **M-Mode 退朝**：从 M-Mode 中断/异常中返回，并伴随特权级降级。 | Bootloader (`start.c`) 初始化完成后，降级跳入 `main.c`。 |
| **`sret`** | Supervisor Return | **S-Mode 退朝**：从 S-Mode 异常中返回，通常降级回 U-Mode。 | `trap.S` 恢复完所有寄存器现场后，执行 `sret` 回到被挂起的 App。 |

---

## 二、 中断控制三剑客：状态、使能与等待 (mstatus / mie / mip)

这三个寄存器构成了 RISC-V 决定“要不要理会这个中断”的终极逻辑门。**(注：以下寄存器在 S-Mode 下有对应的缩水版：`sstatus`, `sie`, `sip`)**。

### 1. `mstatus` (Machine Status) - 系统的“总闸”

它控制着 CPU 当前最核心的状态，包括特权级和中断全局开关。

* **MIE (Bit 3)**：M-Mode 的全局中断使能总闸。为 0 时，CPU 聋了，屏蔽一切 M-Mode 中断。
* **SIE (Bit 1)**：S-Mode 的全局中断使能总闸。
* **MPP (Bits 11:12)**：记录执行 `mret` 时，CPU 应该降级到哪个模式 (11 为 M, 01 为 S, 00 为 U)。

### 2. `mie` (Machine Interrupt Enable) - 中断的“分闸”

即便总闸开了，还得看各个子通道的分闸开没开。

* **MTIE (Bit 7)**：允许 M-Mode 接收定时器中断。
* **STIE (Bit 5)**：允许 S-Mode 接收定时器中断。
* **MSIE / SSIE (Bit 3 / 1)**：允许 M / S 模式接收软件中断。

### 3. `mip` (Machine Interrupt Pending) - 中断的“门铃”

门外有没有人敲门，全看这个寄存器里对应的位有没有变成 `1`。

* **MTIP / STIP**：为 1 时，表示定时器闹钟响了。
* **MSIP / SSIP**：为 1 时，表示收到了软件呼叫。

> **💡 秘书的把戏**：CLINT 硬件闹钟只会把 `MTIP` 置为 1。我们在 M-Mode 写的秘书代码，其实就是手动把 `mip` 里面的 `SSIP`（S-Mode 软件中断等待位）强行改成 1，从而“骗” S-Mode 说有中断来了。

## 三、 硬件计时器：CLINT 专属内存映射寄存器

与普通的 CSR（用 `csrw` 操作）不同，RISC-V 的核心局部中断器 (CLINT) 是外设硬件。**访问它们不能用 `csrw`，必须像访问普通数组一样，用 C 语言的物理内存指针去读写。**

| 物理寄存器 | 宽度 | 作用说明 | 操作系统实战意义 |
| :--- | :--- | :--- | :--- |
| **`mtime`** | 64 位 | **绝对时钟**。开机后以固定频率无限自增的秒表。只读不可写。 | 获取系统当前运行了多久（Tick数）。 |
| **`mtimecmp`** | 64 位 | **闹钟对比器**。硬件会自动对比 `mtime >= mtimecmp`，一旦满足，立刻引爆定时器中断！ | 操作系统实现时间片轮转的核心。每次闹钟响后，向其中加上一个 Interval 设定下一次闹钟。 |

> **⚠️ 32位系统读写 64 位寄存器的死亡陷阱**：
> 在 RV32 中，读取 `mtime` 必须分两次读（先读高 32 位，再读低 32 位）。如果在读的过程中 `mtime` 发生了进位（比如低位从 `0xffffffff` 变成了 `0x00000000`），会导致读出的时间完全错乱。因此必须加入 `do-while` 循环校验高位是否发生变化。

## 四、 异常处理现场四件套 (The Trap Context CSRs)

当突发事件发生时，硬件会自动向这四个寄存器写入案发信息，供 `trap_handler` 处理。

| 寄存器 (S-Mode) | 全称 | 大白话解释 |
| :--- | :--- | :--- |
| **`stvec`** | Trap Vector Base | **总接线员地址**。里面存着 `trap_vector` 汇编代码的绝对物理内存地址。 |
| **`sepc`** | Exception PC | **案发地址存档点**。发生异常时，硬件自动将那条指令的地址存入此处。 |
| **`scause`** | Cause | **案情鉴定书**。最高位决定是异常还是中断，低位代表具体的错误码（如 8代表 ecall，5代表定时器等）。 |
| **`stval`** | Trap Value | **补充证据**。比如发生非法内存访问越界时，这里面存的就是那个罪魁祸首的非法内存地址。 |

---

## 五、 权力下放条约 (Delegation Registers)

默认情况下，所有的 Trap 都会强行进入 M-Mode。为了让 S-Mode 实现真正的自治（高性能处理系统调用和缺页），M-Mode 必须在开机时写入以下寄存器，签署放权协议。

* **`medeleg` (Machine Exception Delegation)**：异常委派。写入 `0xffff` 代表将缺页、非法指令、ecall 等异常交由 S-Mode 的 `stvec` 处理。
* **`mideleg` (Machine Interrupt Delegation)**：中断委派。写入 `0xffff` 代表将外部中断、S-Mode 定时器中断交由 S-Mode 处理。
# RISC-V 核心寄存器解密：mscratch (机器模式暂存寄存器)

> **一句话总结**：`mscratch` 是 RISC-V 硬件专门给 M-Mode (最高特权级) 配备的**“隐秘口袋”**。它平时什么都不干，专门用来存放一个**绝对安全的内核级内存指针**，以便在危机爆发（触发异常/中断）的瞬间，为接线员提供一个安全的落脚点。

---

## 一、 名字的由来：什么是 Scratch？

* **m**：代表 Machine Mode（机器模式，最高特权级）。
* **scratch**：字面意思是“抓痕、草稿纸”。在计算机底层语境下，它指的是**“暂存区”或“便签本”**。

所以，`mscratch` 就是**M 模式专属的暂存寄存器**。除了 M 模式，任何人（包括 S 模式的操作系统和 U 模式的用户程序）都看不见它，更摸不到它。

---

## 二、 为什么要设计它？（核心痛点：零信任模型）

想象一下，CPU 正在执行一个带有漏洞的 U-Mode 用户程序，它的栈指针 `sp` 被破坏了，指向了一个非法的地址。就在这时，定时器中断（闹钟）响了，CPU 强行切入 M-Mode 的汇编接线员代码。

接线员上班的第一件事是**保护现场（保存寄存器）**。但是：
1. 接线员**不能用当前的 `sp`**，因为它是错的，写入会导致系统崩溃甚至被黑客利用。
2. 接线员也**不能直接把数据写死在某个固定的物理地址**，因为未来如果是多核 CPU，每个核心都需要自己独立的保存区，写死地址会导致多个核心互相踩踏。
3. 接线员**不敢使用任何通用寄存器（如 `a0`、`t0`）**去计算内存地址，因为这些通用寄存器里装的都是案发现场的数据，你一用，现场就被破坏了！

**这是一个逻辑死锁：你要保存现场，就需要内存指针；你要找内存指针，就得动用寄存器；你一动寄存器，现场就被破坏了。**

---
